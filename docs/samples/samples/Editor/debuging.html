VEX Engine: Debugging & Crash Analysis ​

This guide explains how to find and interpret error logs for your game. The way errors are reported depends on the Build Configuration (Debug, Release, or Distribution) you selected in the VEX Editor, as well as your operating system.

1. The Log File: game_session.log ​

Every time the game runs, the engine records events to game_session.log.

Content: Standard game logs (e.g., "Level Loaded") followed by a Crash Report if the game closed unexpectedly.

  • Location: <GameFolder>/Engine/logs/game_session.log

2. Build Configurations ​

The amount of detail you get in a crash report depends on how you built the game.

A. Debug ​

  • Best for: Daily coding and logic testing.
  • Behavior: Slower, but safer. Includes extra graphics validation.
  • If it crashes: You don't need to do anything special. The engine automatically prints the error, filename, and line number directly to the console and log file.

B. Release ​

  • Best for: Performance testing.
  • Behavior: Fast and optimized.
  • If it crashes: The log shows raw memory addresses (e.g., 0x7ff973f7518b) instead of line numbers. You must use the Crash Decoder to read them.
  • Symbols: Debugging symbols are stored in Build/Symbols/Release/Latest.

This folder is overwritten every time you build Release mode.

C. Distribution ​

  • Best for: Shipping to players (Steam, Itch.io).

  • Behavior: Maximum speed, minimal logging.

  • If it crashes: Produces raw memory addresses, just like Release.

  • Symbols: The engine automatically archives symbols into a timestamped folder: Build/Symbols/Distribution/YYYY-MM-DD_HH-MM-SS.

You must keep the Symbol folder that matches every version of the game you release. If you lose this folder, you cannot decode player crash reports for that version.

3. How to Decode a Crash (Release & Distribution) ​

To turn "gibberish" hex codes into readable filenames and line numbers, you must use the VexCrashDecoder. The process differs slightly by platform.

Windows Workflow ​

On Windows, the decoder needs the Log, the Symbols, and the Game Folder (to verify binary signatures).

The Command:.\VexCrashDecoder.exe <PathToLog> <PathToSymbols> <PathToGameFolder>

Example: You received a log from a player using the "Halloween Update" (Distribution build).

  1. Log: Where you saved it game_session.log.
  2. Symbols: Located in Build\Symbols\Distribution\2025-10-31_12-00-00.
  3. Game Folder: Located in Build\Distribution.

NOTE

As said before, the best practice is to Archive your public builds. Example would be Archive\MyGame\Dist\HalloweenUpdate. With sub folders Binaries, Symbols.

powershell
.\VexCrashDecoder.exe "C:\Temp\game_session.log" "Build\Symbols\Distribution\2025-10-31_12-00-00" "Build\Distribution"

Linux Workflow ​

On Linux, the decoder only needs the Log and the Symbols. The binary data is not required because the symbols are split into standalone .debug files.

The Command:vexcrashdecoder <PathToLog> <PathToSymbols>

Note: vexcrashdecoder is usually available as a global command. If it is missing or corrupted, check your environment variables or run it directly from the engine tools directory.

Example:

bash
vexcrashdecoder "/home/user/temp/game_session.log" "Build/Symbols/Distribution/2025-10-31_12-00-00"
